*Jordan had been manually testing the rebuilt Meet the Team page — tabbing through the doctor cards, checking that each name was announced as a heading, verifying focus moved correctly after deleting a doctor. It was thorough but slow.*

*"I'm catching issues our research participants described," she said. "But I can't do this for every pull request."*

*"You shouldn't have to," Priya said. "Half the issues you're catching — missing labels, broken ARIA attributes, contrast — a tool could have flagged before the code was even committed. That frees you up to focus on the things only a human can test: does the focus flow make sense? Is the screen reader experience coherent? Those are the issues our users told us about."*

*"So the tools catch the structural stuff," Jordan said, "and I focus on the experiential stuff."*

*"Exactly," Priya said. "The research participants gave us a clear picture of what matters. Let's make sure the easy stuff never reaches your desk."*

## The Accessibility Toolchain

Accessibility tools operate at different layers of the development process, and each layer catches different types of issues. Understanding what each tool can and can't do helps you build an effective pipeline without false confidence.

Select any tool below to learn how to set it up and what it catches.

### [Static Analysis: eslint-plugin-jsx-a11y](/tutorial/dev-tools/eslint-jsx-a11y)

Catches accessibility issues in your JSX at write time — in the editor, before the code even runs. Flags missing alt text, invalid ARIA attributes, non-interactive elements with click handlers, and more.

**What it catches:** Issues visible in the source code — missing attributes, incorrect ARIA usage, non-semantic element choices.

**What it misses:** Anything that depends on rendered output, browser behavior, or visual appearance.

[Learn more about eslint-plugin-jsx-a11y &rarr;](/tutorial/dev-tools/eslint-jsx-a11y)

### [Component Testing: vitest-axe](/tutorial/dev-tools/vitest-axe)

Runs axe-core against your rendered React components during unit tests. Catches issues that only appear after rendering — dynamic content, computed ARIA attributes, generated HTML.

**What it catches:** Issues in the rendered DOM — invalid HTML, missing names, color contrast violations in computed styles.

**What it misses:** Keyboard behavior, focus management, visual layout issues, screen reader compatibility.

[Learn more about vitest-axe &rarr;](/tutorial/dev-tools/vitest-axe)

### [Browser DevTools: Accessibility Tree](/tutorial/dev-tools/browser-devtools)

Chrome, Firefox, and Safari all expose an accessibility tree — the structure that assistive technologies actually see. Inspecting it shows you what a screen reader will announce for any element on the page.

**What it catches:** The gap between what you see and what a screen reader sees — missing names, incorrect roles, hidden content that shouldn't be, visible content that's been accidentally hidden.

**What it misses:** Nothing inherently, but it requires manual inspection. It's a diagnostic tool, not an automated check.

[Learn more about browser DevTools &rarr;](/tutorial/dev-tools/browser-devtools)

### [End-to-End Testing: Playwright + axe](/tutorial/dev-tools/playwright-axe)

Runs axe-core against fully rendered pages in a real browser during end-to-end tests. Catches issues that only appear in the complete application context — after routing, data fetching, and client-side rendering.

**What it catches:** Issues in the fully assembled page — the same things vitest-axe catches, but in the real application context with real data and navigation.

**What it misses:** Keyboard workflows, visual regressions, screen reader behavior. It tests the DOM, not the user experience.

[Learn more about Playwright + axe &rarr;](/tutorial/dev-tools/playwright-axe)

### [Component Libraries: Storybook a11y Addon](/tutorial/dev-tools/storybook-a11y)

If your team uses Storybook for component development, the a11y addon runs axe-core against every story automatically. It surfaces issues during component development — before components are integrated into pages.

**What it catches:** Component-level accessibility issues in isolation — the same things axe-core catches, but surfaced during development rather than in CI.

**What it misses:** Integration issues — how components interact, focus flow between components, page-level structure.

[Learn more about Storybook a11y addon &rarr;](/tutorial/dev-tools/storybook-a11y)

**The essential starter kit:** eslint-plugin-jsx-a11y + vitest-axe + browser DevTools. These three cover static analysis, rendered component testing, and manual inspection with minimal setup.

**Going deeper:** Add Playwright + axe for CI-level coverage, and Storybook a11y addon if your team uses Storybook for component development.

## What Automated Tools Can't Do

Automated accessibility testing catches roughly 30–50% of WCAG issues. The remaining 50–70% requires human judgment:

- **Is the alt text meaningful?** — A tool can tell you an image has alt text. It can't tell you whether "IMG_4382" is useful.
- **Does the keyboard flow make sense?** — A tool can verify elements are focusable. It can't tell you whether the focus order is logical.
- **Is the content understandable?** — A tool can check reading level. It can't tell you whether error messages actually help users recover.
- **Does the screen reader experience make sense?** — A tool can verify ARIA attributes. It can't tell you whether the overall experience is coherent.

This is why automated tooling and [manual testing](/tutorial/testing) are both essential. The tools catch what they can, freeing human testers to focus on what requires judgment.

## Key Takeaways

- Accessibility tools operate at different layers — static analysis, rendered components, full pages, and manual inspection. Each catches different issues.
- No single tool catches everything. Automated tools cover roughly 30–50% of issues.
- Start with eslint-plugin-jsx-a11y, vitest-axe, and browser DevTools. Add Playwright + axe and Storybook a11y for deeper coverage.
- Automated tools catch structural issues (missing labels, broken ARIA, low contrast). Human testing catches experiential issues (is the flow logical, is the content clear).
- The goal of automation is to free human testers from catching issues a computer could have found.
