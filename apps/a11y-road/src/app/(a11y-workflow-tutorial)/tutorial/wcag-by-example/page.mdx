# WCAG by Example

---

*Jordan had just added alt text to every image on the dashboard. "Done. All images have alt attributes. We pass 1.1.1."*

*Priya pulled up a screen reader and tabbed to the first image. It announced: "Image. DSC_4027.jpg." She moved to the next. "Image. photo." And the next. "Image. banner-final-v2."*

*"Technically those aren't empty," Jordan said slowly.*

*"Technically they pass the linter," Priya agreed. "But a screen reader user just heard three file names and learned nothing. The criterion says 'text alternative that serves the equivalent purpose.' These don't."*

*Marcus leaned in. "So the standard is a floor, not a ceiling?"*

*"Exactly," Priya said. "Passing the check is step one. Making it actually useful is the goal. It's like keyboard access — you can make every element reachable with Tab, but if it takes forty keystrokes to reach the main content, you've met the letter of 2.1.1 and missed the point."*

*Ava opened her notebook. "So what would help us? Something that shows each criterion, what it actually looks like when it's done right, and how we catch it when it's not?"*

## A Practical Way to Read WCAG

The [Intopia WCAG 2.2 Map](https://intopia.digital/wp-content/uploads/2023/12/Intopia-WCAG-2.2-Map-Landscape-version.pdf) is an excellent visual reference that organizes every success criterion by principle and guideline — the way WCAG itself is structured. It answers the question: **"What does WCAG say?"**

This page answers a different question: **"What does each criterion look like in practice?"**

For every WCAG 2.2 success criterion, this table shows who benefits, how to check for it, what a real example looks like, and how to catch and prevent regressions. Many criteria help more than one user group — that's not a mistake. It reflects how interconnected accessibility is. Fixing contrast helps low vision users *and* people with color vision differences. Good heading structure helps screen reader users *and* people with cognitive differences.

Conformance levels are marked: **A** (minimum), **AA** (standard target), **AAA** (aspirational). Criteria new in WCAG 2.2 are marked with ✱.

### Beyond the standard

WCAG defines the *minimum* bar for accessibility, not the finish line. Many criteria can be met technically while still leaving users with a poor experience. A few examples:

- **Alt text** (1.1.1) — an image with `alt="IMG_3841.jpg"` passes automated checks but tells a screen reader user nothing. Meaningful alt text describes the *purpose* of the image, not just its presence.
- **Keyboard access** (2.1.1) — every interactive element might be reachable via Tab, but if a user needs forty keystrokes to reach the main content, the experience is frustrating. Efficient keyboard flows matter as much as basic reachability.
- **Captions** (1.2.2) — auto-generated captions technically exist, but garbled output is worse than no captions for someone relying on them. Accurate, well-timed captions are the real goal.
- **Error messages** (3.3.1) — "Invalid input" next to a form field satisfies the criterion, but "Please enter a date in MM/DD/YYYY format" actually helps someone fix the problem.

As you work through the table below, keep this in mind: each criterion is a starting point. The examples in the table aim to show what *good* looks like, not just what passes.

## How to Read This Table

Each table below covers one WCAG 2.2 principle. The columns are:

- **WCAG SC** — The success criterion number, title, and conformance level (A, AA, or AAA). Criteria new in WCAG 2.2 are marked with ✱.
- **How to check** — A plain-English description of what to look for.
- **Who benefits** — The user groups that benefit most. Many criteria help more than one group.
- **Example** — A concrete scenario that shows what passing or failing looks like.
- **How to catch** — The tools and methods that can detect failures.
- **Regression prevention** — For items that require manual testing, a suggested automated test strategy to prevent regressions once the issue is fixed.

### User group key

Each group has a **quick check** — a single question to keep in mind while reviewing.

- **Screen reader** — People who use screen readers (blind or low-vision users). *Quick check: Can someone understand this without seeing it?*
- **Keyboard** — People who navigate with keyboard only. *Quick check: Can I complete the entire task with Tab, Enter, and arrow keys?*
- **Low vision** — People with low vision who zoom, magnify, or customize display. *Quick check: Does the page still work at 200-400% zoom?*
- **Color vision** — People with color vision differences. *Quick check: If I remove color, does the meaning remain clear?*
- **Cognitive** — People with cognitive, learning, or neurological differences. *Quick check: If someone is confused, does the interface help them recover?*
- **Hearing** — People who are deaf or hard of hearing. *Quick check: Can someone use this without hearing the audio?*
- **Motor** — People with motor or mobility impairments. *Quick check: Can someone with limited movement still complete the task?*
- **Vestibular** — People with vestibular disorders or motion sensitivity. *Quick check: Does anything move, flash, or animate without the user asking for it?*

### Detection method key

- **Linter** — eslint-plugin-jsx-a11y or similar static analysis
- **axe** — axe-core (via Playwright, vitest-axe, or browser extension)
- **DevTools** — Browser developer tools (accessibility tree, contrast checker, etc.)
- **Screen reader** — Manual testing with a screen reader (VoiceOver, NVDA, JAWS)
- **Inspection** — Visual or code review
- **axe DevTools** — axe DevTools browser extension (includes guided tests beyond free axe-core)

## Summary

WCAG 2.2 contains **87 success criteria** across four principles:

- **Perceivable** — 29 criteria covering text alternatives, captions, contrast, and adaptability
- **Operable** — 32 criteria covering keyboard access, timing, seizure safety, navigation, and input
- **Understandable** — 21 criteria covering readability, predictability, and input assistance
- **Robust** — 3 criteria (after 4.1.1 was deprecated) covering compatibility with assistive technologies

Of these, **9 criteria are new in WCAG 2.2** (marked with ✱), and **1 criterion (4.1.1 Parsing) was deprecated**.

For most projects, **A and AA conformance** (50 active criteria) is the standard target. AAA criteria are aspirational goals that improve the experience for many users but may not be achievable for all content types.

This table and the [Intopia WCAG 2.2 Map](https://intopia.digital/wp-content/uploads/2023/12/Intopia-WCAG-2.2-Map-Landscape-version.pdf) complement each other:

- **Intopia map** — start with WCAG's structure, find the criteria you need to meet. Best for compliance reviews and audits.
- **This table** — start with a criterion and see what it looks like in practice — who it helps, how to check for it, and how to prevent regressions. Best for development, testing, and code review.

Use the Intopia map when you need to verify conformance. Use this table when you need to understand what *good* looks like and how to build it into your workflow.

And remember: the standard is the floor. The quick checks, the examples, and the regression strategies here are meant to help you go beyond passing and toward something that actually works for the people using it.
